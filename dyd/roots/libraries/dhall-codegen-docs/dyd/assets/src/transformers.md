# Transformers

Transformers turn a `Document.Type` into a new `Document.Type`.

They are used to:

- **normalize** or **lower** certain schema constructs
- **lift** nested records to top-level root definitions (so targets can emit stable named types)

Some renderers call transformers automatically.

## Transformer: lift

**Path:** `dhall-codegen/transformer-lift/package.dhall`

### What it does

The lift transformer walks every root schema and:

1. **Lifts nested records** (records not at depth 0) into separate root definitions.
2. Replaces the original nested record node with a `reference` to the lifted root.
3. Propagates “optionalness” in a way that makes record fields more natural in generated code.

This is what enables renderers to produce stable named types like:

- `ValuesObjectStoreConfigurationWal`

instead of deeply inlining record shapes everywhere.

### Naming

Lifted root names are generated by concatenating path segments.

The segment transformer is configurable:

```dhall
let Lift = ./dhall-codegen/transformer-lift/package.dhall

in  Lift.transform
      Lift.options::{
      , nameSegmentTransformer = ./dhall-codegen/utils/Text/snakeCase.dhall
      }
      myDocument
```

The default `utils/Text/snakeCase.dhall` actually produces **PascalCase** segments:

- `foo_bar` → `FooBar`
- `foo-bar` → `FooBar`
- `objectStore` → `ObjectStore`

### Optional lifting rules

The transformer represents “optionalness” as a separate signal while walking the tree.

Key behaviors:

- **Under a record:** an `optional` value on a required field becomes an optional field (`field?: …`) instead of a nested `optional` wrapper.
- **Through `oneOf`:** if any option is optional, the entire `oneOf` becomes optional.
- **Not through `allOf`, `list`, `set`, `map`, `tuple`:** optional children are re-wrapped with `optional` inside those nodes.

These rules are implemented in the per-node transforms under `dhall-codegen/transformer-lift/transform-*.dhall`.

## Transformer: json-schema lowering

**Path:** `dhall-codegen/transformer-json-schema/package.dhall`

JSON Schema can’t faithfully represent every schema node as-is (notably `map` with arbitrary key types and the `function` node).

This transformer **lowers** those constructs into JSON-schema-friendly forms.

### Map → list of key/value pairs

A `map` becomes a `list` of records with required `key` and `value` fields:

```text
Map<K, V>
  ↓
List<{ key : K, value : V }>
```

This avoids the JSON Schema limitation that object keys are always strings.

### Function → structured record

A `function` becomes a record with:

- `kind`: the literal text "function"
- `input`: a tuple of input types
- `output`: a tuple of output types

```text
Function
  ↓
{ kind = "function", input = (..tuple..), output = (..tuple..) }
```

### How it’s used

The JSON Schema renderer calls this transformer automatically **before** applying the lift transformer.

If you are only rendering to TypeScript/Python/Dhall, you typically do **not** need this transform.
